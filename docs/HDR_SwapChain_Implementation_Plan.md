# HDR Swap Chain — Detailed Implementation Plan

Date: 2026-07-03 | Status: **Ready for execution**

This plan translates the analysis in [HDR_SwapChain_Analysis.md](./HDR_SwapChain_Analysis.md) into a sequenced, file-by-file implementation guide. Each phase is self-contained and can be verified independently.

---

## Overview: What We Are Building

Replace the hardcoded SDR-only swap chain (`RGBA8_UNORM`, `Tonemap_PSMain` with PBRNeutral + sRGB OETF) with an auto-detecting dual-path system:

| Condition | Swapchain Format | Tonemap Shader | Output |
|---|---|---|---|
| HDR display detected + scRGB supported | `RGBA16_FLOAT` | `TonemapHDR_PSMain` (SDR passthrough + Reinhard headroom) | Linear scRGB [0, maxSCRGB] |
| Fallback (no HDR display) | `RGBA8_UNORM` | `Tonemap_PSMain` (PBRNeutral + sRGB OETF) | sRGB [0, 1] |

**Design decisions** from the analysis:
- **Format**: scRGB FP16 (`R16G16B16A16_FLOAT`). No PQ encoding needed.
- **Color space**: `DXGI_COLOR_SPACE_RGB_FULL_G10_NONE_P709`. Set via `SetColorSpace1`.
- **HDR metadata**: NOT needed for scRGB. We skip `SetHDRMetaData` entirely.
- **Display nits**: Queried from `IDXGIOutput6::GetDesc1()` → `MaxLuminance`. Fed to the tonemapping shader.
- **Detection**: Three-layer auto-detection (SDL3 → DXGI color space support → display nits). No user configuration needed.
- **ImGui**: Zero changes — format-agnostic blend states + dynamic `m_SwapchainFormat` propagation.

        if (FAILED(hr))

        // ═══════════════════════════════════════════════════════════════

                m_bIsHDR = false;
                m_SwapchainFormat = nvrhi::Format::RGBA8_UNORM;
                m_MaxDisplayNits = 80.0f;
            }
        }
        else
        {
            m_bIsHDR = false;
            m_SwapchainFormat = nvrhi::Format::RGBA8_UNORM;
            m_MaxDisplayNits = 80.0f;

## Phase 3: HDR Tonemapping Shader

### Task 3.1 — Add `HDRDisplayTonemap()` function and `TonemapHDR_PSMain` entry point

**File**: `src/shaders/Tonemap.hlsl`

**Append after** the existing `Tonemap_PSMain` function (after line ~59):

```hlsl
// ═══════════════════════════════════════════════════════════════
// HDR scRGB Tonemapping
// ═══════════════════════════════════════════════════════════════
// Input:  linear Rec.709 color * exposure (scRGB units)
// Output: linear scRGB values in [0, maxSCRGB]
// 1.0 scRGB = 80 nits (SDR white). Values > 1.0 are HDR.
// SDR content (≤ 1.0) passes through untouched.
// HDR content is compressed via Reinhard rolloff toward display peak.

float3 HDRDisplayTonemap(float3 x, float MaxDisplayNits)
{
    float maxSCRGB = MaxDisplayNits / 80.0;

    float lum = max(x.r, max(x.g, x.b));

    // SDR passthrough: no compression at or below 80 nits
    if (lum <= 1.0)
        return x;

    // Compress only the headroom above SDR white
    float hdrHeadroom = maxSCRGB - 1.0;
    float excess = lum - 1.0;
    float compressedExcess = excess * hdrHeadroom / (excess + hdrHeadroom);

    float newLum = 1.0 + compressedExcess;
    float3 result = x * (newLum / lum);

    // Safety clamp: never exceed display max
    return min(result, maxSCRGB);
}

float4 TonemapHDR_PSMain(FullScreenVertexOut input) : SV_Target
{
    uint2 pixelPos = uint2(input.uv * float2(TonemapCB.m_Width, TonemapCB.m_Height));
    float3 color = HDRColorInput[pixelPos];
    float exposure = ExposureInput[0];

    // Apply exposure
    color = color * exposure;

    float3 hdrOutput = HDRDisplayTonemap(color, TonemapCB.m_MaxDisplayNits);

    // No gamma correction — scRGB is linear
    return float4(hdrOutput, 1.0f);
}
```

**What this does**: Adds the HDR tonemapping pipeline. The existing `Tonemap_PSMain` (SDR path) is completely unchanged. The new `TonemapHDR_PSMain` omits gamma correction because scRGB is linear.

**Key behavior**:
- Values ≤ 1.0 scRGB (≤ 80 nits): **passthrough** — SDR UI and content look identical
- Values > 1.0 scRGB: compressed via Reinhard rolloff toward display peak
- Asymptotically approaches `maxSCRGB` — never harshly clips

**Verification**: Shader compiles without errors via DXC. (Will be verified when shaders are rebuilt in Phase 4.)

## Risk Assessment & Edge Cases

| Risk | Likelihood | Mitigation |
|---|---|---|
| `SDL_GetDisplayForWindow` returns 0 (no display) | Low | Wrap in null check; fall back to SDR |
| `IDXGIOutput6` unavailable (old GPU/driver) | Low | `QueryInterface` failure → fall back to SDR |
| `MaxLuminance` returns 0 or garbage | Low | Sanity check: clamp to [80, 10000] |
| Shader `TonemapHDR_PSMain` fails to compile | Medium | Test with DXC standalone first |
| `PushConstantBytes = 12` breaks SDR path | None | SDR `Tonemap_PSMain` only reads `m_Width` and `m_Height` (first 8 bytes); extra 4 bytes are harmless |
| `RGBA16_FLOAT` swapchain fails to create | Low | Fall back to SDR format |
| `SetColorSpace1` not supported | Low | `IDXGISwapChain4` query failure → fall back to SDR |
| `SaveBackBufferScreenshot` fails with FP16 | Medium | Separate follow-up task; screenshots work with SDR fallback |

3. **HDR calibration integration**: Read Windows HDR Calibration profile for more accurate `MinLuminance` and per-channel primaries.
4. **Alternative tonemapping curves**: Once the system is stable, experiment with Reinhard Extended or ACES Fitted (already documented in the analysis).
